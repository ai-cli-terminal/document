## 1. 프로젝트 개요

### 1.1 배경 — 왜 이 프로젝트가 필요한가

AI CLI 도구가 빠르게 확산되면서 "AI가 명령을 생성하고 곧장 실행한다"는 형태의 터미널이 늘고 있다. 그러나 이런 도구들은 다음과 같은 구조적 문제를 반복한다.

- **잘못된 cwd/컨텍스트 기반 위험 명령 제안**: AI가 현재 작업 디렉터리·git 브랜치·셸 상태를 정확히 추적하지 못한 채 명령을 제안하면, 의도와 다른 경로에서 파괴적 명령(`rm -rf`, 광범위 `chmod` 등)이 실행될 수 있다. 이는 AI 터미널에서 가장 흔한 실패 유형이다.
- **AI 장애의 터미널 전파**: AI 서비스가 느려지거나 죽으면 터미널 자체가 멈추는 설계가 많다. 일상적인 셸 작업까지 AI 가용성에 묶이는 것은 받아들이기 어렵다.
- **민감정보 유출**: 명령·로그·환경 변수에 포함된 Secret/PII가 마스킹 없이 원격 LLM으로 전송되면 자격 증명·개인정보가 외부로 노출된다.

본 프로젝트는 이 세 가지 문제를 정면으로 해결한다. 일반 리눅스 터미널과 **완전 호환**되는 실행 환경을 기본으로 유지하면서, AI 기반 명령 생성·설명·디버깅·로그 분석·자동화를 **안전하게** 결합한다.

설계 철학은 한 문장으로 요약된다.

> **"AI는 명령 실행자가 아니라 의사결정 보조자."**

즉 기본 동작은 기존 터미널과 동일하게 유지하고, AI 기능은 명시적 호출·실행 전 설명·위험도 분석·사용자 확인을 중심으로 제공한다.

이 방향성은 기존 설계 검토(§0.1)에서 확인된 강점 위에 서 있다.

- **"일반 터미널 우선" 원칙의 일관성**: AI는 보조자이며 최종 실행 권한은 사용자에게 있다. AI 장애가 터미널 장애로 전파되지 않는다.
- **실무 수준의 보안·프라이버시 설계**: 위험 명령 분류, Secret/PII 마스킹, Zero-Trust Context Pipeline, 샌드박스 capability drop, 조직 정책 우선순위.
- **상태 일관성 문제의 정면 대응**: Context Consistency Manager와 Built-in 명령 동기화로 잘못된 cwd/컨텍스트 기반 제안을 구조적으로 방지.
- **정적 검사 + 동적 감시 이중화**: 실행 전 정적 Policy Engine과 실행 중 동적 Execution Guardrails Engine을 분리해 방어 깊이를 확보.
- **현실적인 단계 계획**: MVP / MVP+ / Phase 2~4 구분이 명확하고, eBPF·Firecracker·multi-agent 같은 고난도 기능을 후순위로 둠.

> 설계 버전: **v3.3 (MVP spec finalized — ready to build)**. 기준일 2026-06-01.
> 본 계획서는 설계 문서 `→ 00-overview-architecture.md 참조`, `→ 05-roadmap-enhancements-decisions.md 참조`, `→ 06-mvp-implementation-spec.md 참조`의 사실을 기반으로 작성되었다.

### 1.2 핵심 목표

설계 §2.1 기능 목표를 기반으로 한 핵심 목표 10+2개다. 각 목표는 MVP+ 범위(§25.1)와 직접 연결된다.

| # | 핵심 목표 | 한 줄 설명 |
|---|---|---|
| ① | 일반 리눅스 명령어 실행 | PTY 기반으로 bash/zsh와 완전 호환되는 실행 환경 제공 |
| ② | AI CLI 전용 명령 실행 | `ai "..."` 인라인 호출로 별도 모드 전환 없이 AI 사용 |
| ③ | 자연어 기반 명령어 생성 | 자연어 요청을 셸 명령으로 변환하여 제안 |
| ④ | 실행 전 명령어 설명 및 위험도 분석 | 실행 전 명령의 의미·영향·위험도(0~100)를 설명 |
| ⑤ | 명령 실행 결과 요약 | 출력·종료 코드를 사람이 읽기 쉽게 요약 |
| ⑥ | 로그·에러·스택트레이스 분석 | 직전 에러 분석(`ai explain last-error`)으로 원인·해결책 제안 |
| ⑦ | 반복 작업 자동화 스크립트 생성 | 반복 작업을 스크립트로 생성(자동 실행은 금지) |
| ⑧ | 프로젝트 컨텍스트 기반 명령 추천 | cwd·git·셸 상태를 동기화한 컨텍스트 기반 추천 |
| ⑨ | 세션 기록 및 재현 가능한 작업 로그 저장 | 단일 `ai-terminal.db`에 세션·명령·감사 로그 저장 |
| ⑩ | 권한 필요 명령 확인 절차 제공 | 위험도·정책 프로파일에 따른 확인/차단 게이트 |
| ⑪ | 파일 변경 명령 preview/dry-run/diff | 임시 복사본에서 실행 후 diff 확인 → 적용 |
| ⑫ | 실행 전 정적 정책 검사 + 실행 중 동적 가드레일 | Policy Engine(정적) + Guardrails Engine(동적) 이중화 |

> 비기능 목표(§2.2): 높은 셸 호환성, 낮은 입력 지연(일반·AI 경로 완전 분리), 민감 정보 보호(Secret/PII 마스킹 필수), 로컬 우선·오프라인 기본 동작, **AI 실패 시 끊김 없는 터미널 사용**, PTY 상태와 AI 컨텍스트 상시 동기화.

### 1.3 대상 사용자 — 역할별 주요 기능

설계는 4개 페르소나를 정의한다. 각 역할의 대표적 사용 맥락과 주요 기능은 다음과 같다.

| 역할 | 페르소나 | 주요 관심사 | 대표 기능 |
|---|---|---|---|
| **DEV** | 개발자 | 빠른 명령 작성·디버깅, 낮은 지연 | 자연어→명령 변환, 에러 분석, preview/diff, undo, 세션 히스토리 |
| **OPS** | 운영자(SRE) | 안전한 운영, 장애 분석, 위험 명령 통제 | 로그·스택트레이스 분석, 위험도 분석, 강한 확인 게이트, 감사 로그 |
| **RESEARCH** | 연구자 | 반복 실험 자동화, 재현성 | 자동화 스크립트 생성, 사용량·토큰 추적, 재현 가능한 작업 로그 |
| **SECADMIN** | 보안·조직 관리자 | 정책 강제, 마스킹, 감사 추적 | 정책 프로파일(balanced/paranoid), Secret/PII 마스킹, 감사 이벤트 |

> 정책 프로파일: `balanced`(기본), `paranoid`는 MVP 필수 / `poweruser`, `dev`는 Phase2.

### 1.4 프로젝트 범위

MVP 통합 확정값(§30.3 / §31.13) 요약:

```text
Shell    : Hook 기반 기본 + Native Wrapper fallback (rc 자동 수정 금지)
Daemon   : 없음 (파일 락 + SQLite WAL + append-only 로그)
Local LLM: 선택, 7B coder Q4/Q5 baseline, 정책/검증 우회 불가
Indexing : 기본 비활성 (ripgrep keyword + SQLite FTS + recent cache)
Provider : 최소 인터페이스 + capability map (tool-use 추상화 제외)
Undo     : best-effort 파일 롤백만 (500MB 상한, TTL 7일)
Policy   : 로컬 user policy, balanced/paranoid, 로컬 정책 우선
Risk     : rule-based 0~100 점수, Critical 차단, AI 분류는 보조
Preview  : 가능한 파일 변경 명령에 필수, dry-run 우선, 임시 복사본 diff
Privacy  : Secret/PII 마스킹 기본 on, 마스킹 실패 시 원격 AI 차단
Guardrails: static analysis + preview + timeout baseline + capability matrix
```

### 설계 원칙 (판단 기준 17개)

설계·구현·리뷰 시 판단 기준으로 사용하는 17개 핵심 원칙이다(§3). 본 계획서의 모든 결정은 이 원칙에 종속된다.

1. 일반 쉘 호환성이 AI 기능보다 우선한다.
2. AI는 명령 실행자가 아니라 보조자이며, 최종 실행 권한은 사용자에게 있다.
3. AI 기능 장애는 터미널 장애로 전파되지 않아야 한다.
4. PTY 상태와 AI 컨텍스트는 항상 동기화되어야 한다.
5. 로컬·SSH·컨테이너·Kubernetes 컨텍스트는 명확히 구분한다.
6. 모든 외부 컨텍스트는 Zero-Trust Pipeline을 통과한다.
7. Secret/PII 마스킹은 원격 AI 호출 전 필수이다.
8. 원격 AI 호출 전 필요한 최소 정보만 전송한다 (Data Minimization).
9. 위험도 판단과 정책 평가는 로컬에서 먼저 수행한다.
10. 파일 변경 명령은 preview·dry-run·diff를 우선 제공한다.
11. AI 생성 명령은 기본적으로 자동 실행하지 않는다.
12. 정책 프로파일로 사용자·조직 보안 수준을 조정한다.
13. 샌드박스는 보조 안전장치이지 완전한 보안 경계가 아니다.
14. 실행 전 정적 정책 검사와 실행 중 동적 가드레일을 함께 사용한다.
15. 로컬 LLM·인덱싱은 터미널 응답성을 저해하지 않아야 한다.
16. 감사 로그는 추적 가능성을 높이되 민감 정보는 저장하지 않는다.
17. MVP에서도 상태 일관성·마스킹·타임아웃·preview·정책 프로파일은 반드시 포함한다.

> Phase 2 이후 추가되는 서브시스템 원칙 18~20(스킬/MCP/리모트의 Zero-Trust·정책 경계·E2E)은 `→ 00-overview-architecture.md 참조` §3 참조.

## 3. 성공 지표 (KPI)

정량 목표다. 모든 수치는 MVP 수용 기준 및 설계 KPI와 일치한다.

| 분류 | 지표 | 목표값 | 측정/근거 |
|---|---|---|---|
| 성능 | 일반 명령 입력 지연 | **≤ 10ms** | 일반 셸 경로가 AI 계층을 거치지 않음 |
| 성능 | AI 라우팅 지연 | **≤ 100ms** | Classifier → 경로 분기 |
| 성능 | 짧은 AI 응답 | **≤ 3s** | 명령 생성·설명 등 단문 응답 |
| 보안 | Secret/PII 마스킹 누락 | **0건** | 마스킹 실패 시 원격 AI 차단(fail-closed) |
| 보안 | Critical 명령 차단 | **100%** | balanced/paranoid 모두 Critical 차단 |
| 보안 | 위험도 점수 결정성 | **deterministic** | 동일 명령·환경 → 동일 0~100 점수 |
| 품질 | 핵심 모듈 테스트 커버리지 | **≥ 80%** | `→ 11_테스트_전략서.md 참조` |

### 위험도 등급 기준 (0~100)

위험도 점수는 deterministic하며 로컬 규칙 점수가 AI 분류보다 우선한다(§31.4).

| 등급 | 점수 | 설명 |
|---|--:|---|
| Low | 0~24 | 읽기 중심, 부작용 없음 |
| Medium | 25~49 | 제한적 파일 변경 / 프로세스 제어 |
| High | 50~79 | 광범위 변경 · 권한 상승 · 네트워크 실행 |
| Critical | 80~100 | 시스템 손상 · 대량 삭제 가능 |
