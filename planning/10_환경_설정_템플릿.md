# 10. 환경 설정 템플릿
> **프로젝트명**: AI CLI 통합 리눅스 터미널
> **버전**: v1.0
> **작성일**: 2026-06-01
> **기술 스택**: Rust · ratatui · tokio · portable-pty · SQLite (대안: Go)
---

본 문서는 "AI CLI 통합 리눅스 터미널"(설계 v3.3)의 환경 설정 정본이다. 본 도구는 서버형이 아니라 **사용자 머신에 설치되는 단일 바이너리**이므로, 일반적인 local/dev/staging/production 환경 매트릭스 대신 **개발 환경 / CI / 배포 채널** 관점으로 재구성한다.

> 상호참조: 정책 프로파일 전체 필드는 `→ docs/06-mvp-implementation-spec.md §31.3 참조`, 마스킹 룰셋은 `→ §31.8 참조`, 셸 통합 결정은 `→ docs/05-roadmap-enhancements-decisions.md §29.1 참조`.

## 1. 환경 매트릭스

| 구분 | 개발 환경(dev) | CI | 배포 채널(release) |
|---|---|---|---|
| 목적 | 로컬 빌드·디버깅 | 자동 검증·회귀 | 최종 사용자 설치 |
| 빌드 | `cargo build` | `cargo build --release` + lint/test | 서명된 단일 바이너리 + 체크섬 |
| 설정 위치 | `~/.config/ai-terminal/config.toml` | 격리된 임시 `$XDG_CONFIG_HOME` | 사용자 머신의 `~/.config/...` |
| 데이터 위치 | `~/.local/share/ai-terminal/` | 임시 디렉터리(테스트 후 정리) | `~/.local/share/ai-terminal/` |
| AI provider | 로컬 mock 또는 개인 키 | mock/stub(외부 호출 금지) | 사용자 설정 provider |
| 정책 프로파일 | `balanced`(개발 편의) | `paranoid`(회귀 안전) | `balanced`(기본) |
| 원격 AI | 선택 | 차단 | 설정 의존 |
| 샌드박스 | tmpdir | tmpdir/container | container(MVP+) |
| 배포 | 해당 없음 | 아티팩트 업로드 | GitHub Actions + 자동 업데이트 notify |

> CI에서는 실제 외부 AI 호출을 금지하고 mock provider를 사용한다. Golden Set(§22.6)·속성 기반 검증(§29.13)은 결정성(`temperature=0`)을 강제한다.

### 2.1 섹션별 주석 설명

| 섹션 | 역할 | 주의점 |
|---|---|---|
| `[general]` | 기본 셸·AI 모드·테마·히스토리 | `ai_mode`는 inline/dedicated/hybrid |
| `[ai]` | provider·모델·트리거 alias·충돌 전략 | `auto_execute=false` 고정 권장(자동 실행 금지) |
| `[ai.timeout]` | 연결 5s/첫토큰 15s/요청 60s/요약 180s | AI 장애가 셸을 막지 않게 함(§16.2) |
| `[ai.context]` | 토큰 윈도·청킹·압축·에러 우선 | 컨텍스트 수집은 비동기(§20.2) |
| `[ai.usage]` | 토큰/비용 표시·예산 | session $2 / monthly $30, 80% 경고 / 100% 차단 |
| `[ai.auto_healing]` | 1회 분석·재제안 | `auto_execute=false` 항상 권장(§16.3), 자동 재실행 금지 |
| `[ai.cache]` | 정확/시맨틱 캐시 | 캐시엔 마스킹된 컨텍스트만, at-rest 암호화 대상 |
| `[ai.routing]` | 멀티 provider 라우팅·폴백 | `sensitive_context_forces_local=true`가 정책보다 우선 |
| `[security]` | 위험 확인·Critical 차단·마스킹 | `block_remote_ai_on_masking_failure=true`(fail-closed) |
| `[security.pii_masking]` | PII 항목 토글 | 위치는 기본 미마스킹 |
| `[security.retention]` | 보존·암호화·삭제권 | 세션 30일 / 히스토리 365일 |
| `[sandbox]` | 격리 백엔드·리소스 한도 | MVP는 tmpdir/container, 실패 시 실제 실행 자동 전환 금지(§16.2) |
| `[preview]` | diff/dry-run 미리보기 | 파일 수정은 preview 강제 |
| `[hallucination_guard]` | 바이너리/플래그 실재성 | MVP는 바이너리 존재 검증, 플래그 검증은 Phase 2 |
| `[local_llm]` | Phase 2 로컬 LLM | 기본 비활성, 정책/Verification 우회 불가 |
| `[telemetry]` | opt-in OTLP | 기본 비활성, 명령 내용 미전송 |
| `[logging]` | 세션/감사/사용량 로그 | `redact_logs=true` |
| `[skills]` | 통합 스킬(§26) | paranoid에서 signed만 허용 |
| `[mcp]` | MCP 연동(§27) | `auto_connect=false`, mutate/external 컨센트 |
| `[remote]` | 리모트(§28) | 기본 비활성, High/Critical 원격 승인 차단 |

## 3. 셸 hook 설치 (§29.1, §31.1)

rc 파일은 **자동 수정하지 않으며**, 명시적 `ai init shell` 명령으로만 변경한다. MVP 필수 셸은 `bash`·`zsh`.

### 3.1 rc 삽입 블록 (§31.1)

설치 시 rc 파일에 아래 블록만 추가한다. `command -v ai` 가드로 감싸 미설치 환경에서도 무해하다.

```bash
# AI Terminal integration
if command -v ai >/dev/null 2>&1; then
  eval "$(ai shell-hook zsh)"   # bash는 'ai shell-hook bash'
fi
```

### 3.2 설치 UX

```bash
ai init shell             # 설치 (확인 후)
ai init shell --dry-run   # 적용 없이 미리보기만
ai init shell --diff      # 적용 예정 diff 표시
ai init shell --uninstall # AI 터미널이 삽입한 블록만 제거
```

`--diff` 출력 예:

```diff
--- ~/.zshrc
+++ ~/.zshrc
@@ -42,3 +42,8 @@
 export PATH="$HOME/.local/bin:$PATH"
+# AI Terminal integration
+if command -v ai >/dev/null 2>&1; then
+  eval "$(ai shell-hook zsh)"
+fi
```

- `--dry-run`은 파일을 수정하지 않는다.
- `--uninstall`은 삽입 블록만 제거하고 사용자 작성 라인은 절대 수정하지 않는다.
- Hook 미설치/비호환 시 Native Wrapper로 fallback(§29.1, §30-1).

### 3.3 shell_integration 설정 (§29.1)

```toml
[shell_integration]
mode = "hook"             # hook(기본) | wrapper | auto
fallback = "wrapper"      # hook 미설치/비호환 시
auto_install_hooks = false   # rc 자동 수정 금지, 명시적 ai init shell 필요
require_diff_before_install = true
hook_ipc_socket = "~/.local/share/ai-terminal/hook.sock"
```

## 4. 개발 빌드 환경

### 4.1 Rust 툴체인 (1순위, §24.1)

| 항목 | 권장 |
|---|---|
| 툴체인 | stable(rustup), `rustfmt`·`clippy` 컴포넌트 |
| 빌드 | `cargo build` / `cargo build --release` |
| 테스트 | `cargo test`, `cargo nextest`(선택) |
| 감사 | `cargo audit`(의존성 취약점), `cargo deny`(라이선스/공급망) |

`rust-toolchain.toml` 예:

```toml
[toolchain]
channel = "stable"
components = ["rustfmt", "clippy"]
```

주요 crate(§24.1):

```toml
# Cargo.toml 발췌 (버전은 예시)
[dependencies]
ratatui = "*"        # TUI
crossterm = "*"      # 터미널 이벤트
tokio = { version = "*", features = ["full"] }   # async 런타임
portable-pty = "*"   # PTY
serde = { version = "*", features = ["derive"] }
toml = "*"           # 설정 파싱
clap = { version = "*", features = ["derive"] }  # CLI
tracing = "*"        # 구조적 로깅
rusqlite = "*"       # SQLite(WAL)
```

검색/인덱싱은 ripgrep + SQLite FTS(MVP), 벡터 검색(sqlite-vss/LanceDB)·로컬 LLM(Ollama)은 Phase 2.

### 4.2 대안: Go (§24.2)

운영자 친화적 단일 바이너리 배포를 더 중시할 경우의 대안 스택.

| 영역 | 후보 |
|---|---|
| 언어 | Go |
| TUI | bubbletea / lipgloss |
| PTY | creack/pty |
| CLI / 설정 | cobra / viper |
| 로깅 | slog, zap |
| 로컬 DB | SQLite |

### 4.3 빌드/검증 명령

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo test
cargo audit
```

```text

  ai-terminal.db      # SQLite(WAL) — sessions/commands/ai_requests/usage_events/audit_events/context_snapshots/locks
  sessions/           # 세션 JSONL 기록
  logs/               # 세션/감사/사용량 로그

  hook.sock           # Hook IPC 소켓(§29.1)
```
