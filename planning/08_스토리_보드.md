## 1. 페르소나

### P1. 개발자 · 지훈 (DEV, 29세)
- 역할: 백엔드 개발자. Node/Rust 마이크로서비스를 매일 다룬다.
- 상태: 터미널 작업이 많지만 `sed`/`find`/`awk` 복잡한 옵션을 매번 검색한다. 위험한 한 줄 명령으로 파일을 날린 적이 있다.
- 고민: 명령어 조합을 빠르게 만들고 싶지만, 잘못 실행해 코드/파일이 손상될까 두렵다.
- 기대: 자연어로 명령을 생성하되, **실행 전 diff와 위험도를 눈으로 확인**하고 결정하고 싶다.

### P2. 운영자·SRE · 미라 (OPS, 35세)
- 역할: 인프라 운영자. SSH로 여러 prod 서버·컨테이너·k8s 파드를 오간다.
- 상태: 장애 대응 시 로그를 수동으로 tail/grep하느라 시간이 든다. 원격 컨텍스트를 로컬로 착각해 잘못된 서버에 명령을 친 경험이 있다.
- 고민: 지금 내가 **어느 환경(local/ssh/container/k8s)에 있는지** 항상 명확해야 하고, 위험 명령은 반드시 막혀야 한다.
- 기대: 에러를 AI가 분석·요약하고, 컨텍스트가 프롬프트에 또렷이 표시되며, 외출 중에도 모바일로 승인하고 싶다.

### P3. 연구자 · 다니엘 (RESEARCH, 41세)
- 역할: 데이터/ML 연구자. 대용량 로그·실험 결과를 다루며 반복 전처리 작업이 많다.
- 상태: 같은 전처리 스크립트를 반복 작성한다. 사내 정책상 민감 데이터의 외부 전송이 제한된다.
- 고민: 반복 작업을 자동화 스크립트로 만들고 싶지만, 데이터가 외부 AI로 새어 나갈까 우려된다.
- 기대: 로그 요약·스크립트 생성을 받되, **로컬 우선·오프라인에서도 기본 기능**이 동작하길 바란다.

### P4. 보안·조직 관리자 · 성호 (SECADMIN, 38세)
- 역할: 보안·플랫폼 관리자. 팀 전체의 터미널 보안 정책을 책임진다.
- 상태: 개발자들이 시크릿을 명령/로그에 노출하거나 외부 AI에 민감 컨텍스트를 보낼까 걱정이다.
- 고민: 마스킹 누락·정책 우회·원격 승인 남용을 **조직 차원에서 통제**해야 한다.
- 기대: 정책 프로파일·조직 정책으로 강제하고, 마스킹 실패 시 차단되며, 감사 로그로 추적 가능하길 바란다.

## 2. 역할별 Epic → Story → AC

### Epic A (DEV) — 안전한 AI 명령 생성과 미리보기

시나리오 근거: `00-overview-architecture.md` §4.2(명령 생성), §4.5(파일 변경 미리보기).

**Story A-1 — 자연어로 명령 생성**
> DEV로서 **자연어로 쉘 명령을 생성**하고 싶다. 왜냐하면 복잡한 옵션을 매번 검색하지 않고 빠르게 작업하고 싶기 때문이다.

관련: `FR-AI-003`, `FR-AI-002`, `FR-AI-004` · 화면 `SCR-T-AISUG-001`, `SCR-T-AISUG-002`

**AC:**
- [ ] `ai "현재 디렉터리에서 100MB 이상 파일 찾아줘"`로 명령이 생성된다.
- [ ] 생성 중 부분 결과가 스트리밍되며, **완성 전에는 실행할 수 없다.**
- [ ] 생성 완료 시 명령·위험도(등급+0~100 점수)·설명·Next(실행/수정/취소)가 표시된다.
- [ ] 기본(balanced)에서 즉시 실행하지 않고 사용자 확인을 요청한다.

**Story A-2 — 파일 변경 명령 diff 미리보기**
> DEV로서 **파일 변경 전 diff를 미리 보고** 싶다. 왜냐하면 잘못된 치환으로 파일이 손상되는 것을 막고 싶기 때문이다.

관련: `FR-AI-005`, `FR-AI-006`, `FR-GUARD-004`, `FR-STORE-003` · 화면 `SCR-T-PREVIEW-001`, `SCR-T-PREVIEW-002`

**AC:**
- [ ] `ai preview "모든 .txt의 foo를 bar로 바꿔줘"`가 임시 복사본 dry-run으로 diff를 보여준다.
- [ ] dry-run을 지원하는 도구(rsync/git clean/terraform/kubectl)는 도구의 dry-run을 우선 사용한다.
- [ ] Preview Pane에 명령·위험도·영향 파일·요구 권한(FileWrite)·토큰/비용이 표시된다.
- [ ] `Apply? [y/N]`에서 `y`를 선택해야만 실제 경로에 적용되고 before/after 스냅샷이 기록된다.

**Story A-3 — 에러 분석과 자가 치유 제안**
> DEV로서 **직전 실패 명령의 원인 분석과 수정안**을 받고 싶다. 왜냐하면 에러 메시지 해석에 시간을 쓰고 싶지 않기 때문이다.

관련: `FR-AI-007`, `FR-AI-008` · 화면 `SCR-T-AISUG-003`

**AC:**
- [ ] `ai fix last-error`가 종료 코드·stderr·cwd·관련 파일을 바탕으로 원인을 설명한다.
- [ ] 수정 명령은 1회 자동 *제안*만 하고, **자동 재실행하지 않는다.**
- [ ] High 이상·sudo·삭제·네트워크 전송·예산 초과 시 자가 치유가 비활성화된다.
- [ ] 제안 실행 여부는 `Run suggested fix? [y/N]`로 사용자가 결정한다.

### Epic B (OPS) — 컨텍스트 인지와 위험 명령 보호

시나리오 근거: §4.3(에러 분석), §4.4(위험 명령 보호), §4.6(원격/컨테이너 컨텍스트).

**Story B-1 — 실행 컨텍스트 명확화**
> OPS로서 **지금 어느 환경에서 명령을 실행하는지** 프롬프트에서 즉시 알고 싶다. 왜냐하면 prod 서버와 로컬을 혼동하면 사고로 이어지기 때문이다.

관련: `FR-CTX-002`, `FR-CTX-001`, `FR-CTX-003`, `FR-CTX-005` · 화면 `SCR-T-SHELL-002`, `SCR-T-SHELL-003`

**AC:**
- [ ] 프롬프트가 `[ssh:prod-web-01:/var/www/app]$`처럼 local/ssh/container/k8s를 명시한다.
- [ ] `cd`·`source`·SSH 접속 등 상태 변경 후 AI 컨텍스트가 자동 동기화된다.
- [ ] AI 컨텍스트 cwd와 실제 cwd가 불일치하면 갱신 확인을 요청한다.

**Story B-2 — 로그·에러 분석 (원격)**
> OPS로서 **원격 서버 로그를 AI로 요약·진단**하고 싶다. 왜냐하면 장애 시 빠르게 원인을 좁히고 싶기 때문이다.

관련: `FR-AI-010`, `FR-AI-007`, `FR-SHELL-005`, `FR-AI-009` · 화면 `SCR-T-HYBRID-001`, `SCR-T-AISUG-001`

**AC:**
- [ ] `ai shell`에서 자연어 질문과 `!tail -n 100 ...`을 혼합해 사용할 수 있다.
- [ ] 대용량 로그는 슬라이딩 윈도우/청크 요약으로 처리한다.
- [ ] 응답 대기 중 `Ctrl+C`가 AI 요청만 취소하고 PTY를 손상시키지 않는다.

**Story B-3 — 위험 명령 차단·실행 중 가드레일**
> OPS로서 **위험 명령이 실행 전·실행 중에 모두 막히길** 원한다. 왜냐하면 단 한 번의 사고가 서비스 장애가 되기 때문이다.

관련: `FR-SEC-001`, `FR-GUARD-001`, `FR-GUARD-002`, `FR-GUARD-003`, `FR-GUARD-005` · 화면 `SCR-T-POLICY-002`, `SCR-T-AISUG-002`

**AC:**
- [ ] `rm -rf /` 같은 Critical 명령은 실행 전 차단된다(차단율 100%).
- [ ] 대량 삭제(1,000개 초과 등) 시 실행이 일시정지되고 사용자 확인을 요구한다.
- [ ] Docker socket·`/etc`·`/proc`·`/sys`·`/dev` 접근 시 프로세스 그룹이 종료되고 인시던트 요약이 표시된다.
- [ ] AI는 destructive 명령보다 안전한 대안(dry-run)을 먼저 제안한다.

**Story B-4 — 모바일 원격 승인 (Phase 3)**
> OPS로서 **외출 중에도 AI 확인 프롬프트를 모바일로 승인/거부**하고 싶다. 왜냐하면 장기 작업 중 자리를 비울 때가 많기 때문이다.

관련: `FR-REMOTE-001`, `FR-REMOTE-002`, `FR-REMOTE-003`, `FR-REMOTE-004` · 화면 `SCR-T-REMOTE-001`

**AC:**
- [ ] 명시적 페어링(QR/일회용 코드)·디바이스별 키로만 연결되고 E2E 암호화가 적용된다.
- [ ] 확인 대기·완료·guardrail 발동·예산 초과 시 푸시 알림을 받는다.
- [ ] High 명령 원격 승인은 옵션이며, **Critical·외부 부작용 MCP 도구는 원격 승인이 금지**된다.
- [ ] 원격 스트리밍 출력에도 Secret/PII 마스킹이 적용되고, 모든 원격 액션이 디바이스 ID와 함께 감사된다.

### Epic C (RESEARCH) — 자동화와 로컬 우선

시나리오 근거: §4.2(명령/스크립트 생성), §4.5(미리보기), 로컬 우선 원칙(§21).

**Story C-1 — 반복 작업 스크립트 생성**
> RESEARCH로서 **반복 전처리 작업을 스크립트로 생성**하고 싶다. 왜냐하면 매번 같은 명령을 손으로 작성하기 때문이다.

관련: `FR-AI-011`, `FR-AI-006`, `FR-STORE-006` · 화면 `SCR-T-AISUG-001`, `SCR-T-PREVIEW-002`

**AC:**
- [ ] `ai script "매일 새벽 로그 압축"`이 스크립트 초안을 생성한다.
- [ ] 생성된 스크립트는 실행 전 preview/diff와 위험도가 표시된다.
- [ ] `ai history`/`ai recall`로 과거 작업을 자연어로 검색해 재사용할 수 있다.

**Story C-2 — 로컬 우선·오프라인 동작과 민감 데이터 보호**
> RESEARCH로서 **민감 데이터를 외부로 보내지 않고 로컬에서** AI 보조를 받고 싶다. 왜냐하면 사내 정책상 외부 전송이 제한되기 때문이다.

관련: `FR-OPS-008`, `FR-AI-018`, `FR-SEC-002`, `FR-SEC-006`, `FR-OPS-006` · 화면 `SCR-T-PREVIEW-003`

**AC:**
- [ ] 민감 정보가 포함되면 라우팅이 로컬 AI를 강제한다(`sensitive_context_forces_local`).
- [ ] 원격 호출 전 전송 범위·마스킹 결과를 컨센트 화면으로 확인한다.
- [ ] 마스킹 실패가 감지되면 원격 호출이 fail-closed로 차단된다.
- [ ] 네트워크가 없거나 AI가 실패해도 일반 터미널 기능은 끊김 없이 동작한다.

**Story C-3 — 로그 요약**
> RESEARCH로서 **대용량 로그를 요약**받고 싶다. 왜냐하면 수십만 라인을 직접 읽기 어렵기 때문이다.

관련: `FR-AI-010`, `FR-AI-013`, `FR-SHELL-007` · 화면 `SCR-T-AISUG-001`

**AC:**
- [ ] `ai summarize /var/log/app.log`가 청크 분할·단계적 요약으로 결과를 반환한다.
- [ ] 대량 출력은 ring buffer로 최근 N라인만 유지하고 AI에는 요약/샘플만 전달한다.
- [ ] 긴 요약은 스트리밍으로 표시된다.

### Epic D (SECADMIN) — 정책·마스킹·거버넌스

시나리오 근거: 보안 원칙(§3 원칙 6~9, 12, 18~20), Zero-Trust(§10.3), 정책 프로파일(§12).

**Story D-1 — 정책 프로파일·조직 정책 강제**
> SECADMIN으로서 **조직 보안 수준을 정책 프로파일로 강제**하고 싶다. 왜냐하면 사용자별 설정 편차가 보안 구멍이 되기 때문이다.

관련: `FR-SEC-004`, `FR-SEC-005`, `FR-SEC-001` · 화면 `SCR-T-POLICY-001`

**AC:**
- [ ] `ai policy set paranoid`로 모든 AI 명령 확인·원격 AI 제한·강한 마스킹이 적용된다.
- [ ] 조직 정책(Org>Team>User>Default)이 readonly로 적용되면 사용자가 완화할 수 없다.
- [ ] balanced/paranoid 프로파일의 권위 필드값(§31.3)이 일관되게 적용된다.

**Story D-2 — Secret/PII 마스킹 보장**
> SECADMIN으로서 **시크릿·PII가 AI·로그·감사 어디에도 평문 노출되지 않길** 원한다. 왜냐하면 단 한 번의 유출이 치명적이기 때문이다.

관련: `FR-SEC-002`, `FR-SEC-006`, `FR-CTX-006`, `FR-STORE-003` · 화면 `SCR-T-PREVIEW-003`, `SCR-T-DOCTOR-001`

**AC:**
- [ ] AI 요청·세션 로그·감사 로그 저장 전 모두 마스킹이 적용된다(누락 0건 목표).
- [ ] 마스킹 전 원문은 메모리에서만 일시 사용되고 디스크에 저장되지 않는다.
- [ ] `.env`·private key·credential 파일은 명시 허용 전까지 컨텍스트에 포함되지 않는다.
- [ ] `ai doctor`에서 secret/pii 마스킹 룰 적재 수를 확인할 수 있다.

**Story D-3 — 감사 추적성과 디버그 번들 안전성**
> SECADMIN으로서 **중요 명령을 추적**하되 민감 정보는 저장하지 않길 원한다. 왜냐하면 추적성과 프라이버시를 동시에 만족해야 하기 때문이다.

관련: `FR-STORE-003`, `FR-OPS-004`, `FR-STORE-004` · 화면 `SCR-T-DOCTOR-001`

**AC:**
- [ ] 파일 삭제·권한 변경·배포 등 중요 명령은 before/after 스냅샷과 함께 감사 로그에 기록된다.
- [ ] git 디렉터리는 `git diff --stat`/`--name-only`만 자동 기록하고 원문 diff는 저장하지 않는다.
- [ ] `ai doctor --export-debug-bundle`은 Secret·PII·원문 로그·`.env`·private key를 제외한다.
- [ ] AI 생성 명령과 사용자 원본 입력이 분리 저장된다.

**Story D-4 — 스킬·MCP·원격 확장의 정책 경계 (Phase 2~3)**
> SECADMIN으로서 **스킬·MCP 도구·원격 제어가 정책 경계 안에서만** 동작하길 원한다. 왜냐하면 추가 능력이 보안 우회 경로가 되어서는 안 되기 때문이다.

관련: `FR-SKILL-004`, `FR-SKILL-005`, `FR-MCP-004`, `FR-MCP-005`, `FR-REMOTE-002` · 화면 `SCR-T-PREVIEW-001`, `SCR-T-AISUG-002`

**AC:**
