## Problem Statement

`planning/`의 17종 산출물과 `docs/`의 v3.3 설계는 "일반 리눅스 터미널과 100% 호환되면서 AI 보조를 안전하게 얹은 터미널"을 그린다. 설계 완성도는 높지만 전부 안전·정책·일관성 얘기이고, **"친구한테 보여줬을 때 와 하는 순간"은 비어 있다.** office-hours에서 그 빈칸을 채우는 한 가지 빌드를 정했다.

학습/재미 프로젝트로서, 이 세션의 목표는 사업성 검증이 아니라 **"동작하는 한 바퀴를 만들고, E2E 암호화·디바이스 페어링·위협 모델을 손으로 구현해보는 것"**이다.

## What Makes This Cool

선택한 "와"는 **C(모바일 승인 푸시) + D(환각 0 컨텍스트 인식)의 결합**이다. 둘은 따로 노는 기능이 아니라 하나의 thesis로 붙는다:

> 터미널이 실제 상태(cwd·branch·git dirty·컨테이너)를 **항상 정확히** 알기 때문에(D), 폰으로 날아오는 승인 요청(C)이 **믿을 만하다**. 원격 승인의 진짜 적은 "내가 지금 뭘 승인하는지 모름"인데, 컨텍스트 정확도가 그 적을 없앤다. **D가 C를 안전하게 만든다.**

데모 장면: 긴 빌드/배포를 돌려놓고 외출 → 위험 명령 지점에서 폰이 진동 → 화면에 *그 머신의 진짜 상태*와 위험도 0~100 분해가 뜸 → Approve/Reject. "집 나가 있어도, 폰에 뜬 그 화면이 진짜 그 머신의 진짜 상태다."

## Constraints

- 학습/재미 프로젝트 — 첫 데모까지의 속도보다 "제대로 배운다"를 우선(사용자가 D3에서 풀 E2E를 명시적으로 선택).
- 기존 설계의 정본값을 재료로 재사용: §31.10 컨텍스트 필수 필드, §31.4 위험도 0~100 점수, §10.5 감사 로그, §28.4 보안 정책.
- §28 보안 모델을 지킨다: E2E 암호화, 디바이스 페어링. **원격 승인은 §30-13 정본대로 기본 Medium까지(opt-in), High는 명시적 오버라이드로만, Critical은 어떤 경우에도 폰 승인 불가.** (Premise 5 참조 — "최대 High" 같은 표현은 정본 위반.)
- **payload 마스킹**: 폰에 표시되는 컨텍스트(cwd/git/명령어)는 폰 화면·PWA 캐시 노출 위험이 있으므로 §31.8 Secret/PII 마스킹을 payload 생성 시 적용한다(§28.4 출력 마스킹).
- **키 회전**: 디바이스 키는 revoke+재페어링으로 커버하되, 데몬 측 키·릴레이 채널 토큰 유출 시 회전 경로도 둔다(§28.4 "키 유출 시 회전").
- 스택: Rust + tokio (데몬), 대안 Go. 모바일 수신측은 PWA.

## Premises (모두 동의됨 + 리뷰 반영)

1. **D가 C의 전제다.** 신뢰할 수 있는 원격 승인이 목표이며, 컨텍스트 정확도 없는 폰 승인은 설계가 막으려던 실패("뭘 하는지 모르고 위험한 걸 승인")를 모바일로 확장하는 꼴이다. **따라서 컨텍스트가 stale/mismatch(§31.10 refresh 트리거)면 payload에 stale 경고를 띄우거나 승인을 보류한다 — 이것이 C+D 결합의 실제 증명이다.**
2. **순서 역전을 의식적으로 받아들인다(데몬 충돌 포함).** 설계서는 데몬(§30-2 "MVP 데몬 없음")·원격 릴레이/승인(§28, Phase 3)을 명시적으로 뒤로 미뤘다. 이 학습 빌드는 그 결정을 뒤집어 MVP보다 Phase 3 인프라(데몬+원격)를 먼저 짓는 **별도 트랙**임을 코드/문서에 명시한다. 학습 목적이면 "보안을 제대로 배운다"가 보상이고 첫 데모가 늦는 비용을 감수한다.
3. **성공 기준은 사업성이 아니다.** "동작하는 한 바퀴 + E2E/페어링/위협모델 직접 구현"이 성공. 사용자·수익 검증은 범위 밖.
4. **산출물이 2개, 디바이스는 1개.** 호스트 데몬 + 모바일 수신측(PWA). **다중 디바이스·권한 분리(§28.4 read_only/approve/full)는 비범위.**
5. **위험도 경계 — §30-13 정본 결정과의 충돌을 명시적으로 처리한다.** 원본 **§30-13 확정 결정은 "원격 승인은 Medium까지(opt-in), High·Critical 모두 기본 차단"**이다(§28.4의 `allow_remote_approval_for_high_risk=false` 기본을 §30-13이 "차단"으로 확정). 이 빌드의 **기본값은 §30-13대로 Medium까지만 원격 승인, High·Critical은 로컬 전용.** "deploy 승인" 같은 High 데모를 원하면, High 원격 승인은 §30-13을 뒤집는 **명시적 opt-in 오버라이드**로만 켜고(데몬 순서 역전과 동일 수준의 의식적 충돌), Critical은 어떤 경우에도 원격 승인 불가.
6. **재생공격(replay)을 한 바퀴 성공 기준에 포함한다.** payload에 `nonce + approval_id + expires_at`을 필수로 박고, 데몬이 nonce를 1회용으로 소비한다. 만료·재사용 토큰은 거부.
7. **non-extractable 키의 트레이드오프를 받아들인다.** PWA 키는 WebCrypto non-extractable(IndexedDB)로 두어 추출·백업 불가 — 이게 학습 목표이자 보안 장점이지만, 분실/캐시 삭제 시 **유일한 복구는 데몬에서 revoke + 재페어링**이다(별도 키 백업 없음).

## Approaches Considered

### Approach A: 자체 릴레이 + PWA (minimal viable, 풀-E2E 안에서) — 채택
Rust 데몬이 §31.10 컨텍스트 스냅샷을 유지하다가 승인 필요 이벤트에서 *승인 payload*(명령 + 위험도 0~100 분해 + 컨텍스트)를 만든다. 페어링 때 QR로 디바이스 키쌍 발급 → payload를 디바이스 공개키로 암호화 → 자체 WebSocket 릴레이는 **암호문만 중계**(평문 못 봄). 승인 = 서명 토큰 회신. 폰은 PWA.
- Effort: L (human ~1~2주 / CC ~2~3일) · Risk: Med
- Pros: §28 E2E 충실(릴레이가 못 읽음) / PWA라 앱스토어 없음 / 크립토를 손으로 구현 → 학습 목표 정통
- Cons: iOS PWA 푸시 불안정 / 릴레이 1대 self-host 운영 부담 / 웹 크립토 E2E 까다로움
- Reuses: §31.10, §31.4, §28.4, §10.5

### Approach B: 직결(Tailscale/WireGuard) + 네이티브 앱 (ideal architecture)
릴레이 없음. 데몬이 승인 엔드포인트를 사설 메시에서만 노출(§28.2 direct). 네이티브 앱 QR 페어링 + 디바이스 키 + 생체인증 게이트. mTLS 직결. Critical 원격 차단.
- Effort: XL (human ~1~2달 / CC ~1~2주+) · Risk: Med-High
- Pros: §28에 가장 충실 / 보안 학습량 최대 / 네이티브 푸시 신뢰도
- Cons: 네이티브 앱+VPN 세팅으로 첫 진동까지 가장 김 / 동력 소진 위험

### Approach C: 기존 메신저 차용 (creative/lateral)
모바일 클라이언트를 안 만들고 Signal 또는 ntfy+자체암호 한 겹으로 인터랙티브 승인 메시지를 보낸다.
- Effort: M (human ~3~5일 / CC ~1일) · Risk: Med
- Pros: 모바일 코드 0 / 이번 주에 진짜 진동 / E2E 한 겹은 직접 구현
- Cons: 채널 보안 완전 통제 못함(Telegram bot≠E2E) / 버튼 payload 제약 / "내 제품" 느낌 덜함

## Recommended Approach

**Approach A — 자체 릴레이 + PWA.** 풀-E2E를 고른 의도(보안 정통)와 학습 목표(크립토를 손으로)를 둘 다 챙기면서, 네이티브 앱스토어 마찰 없이 진짜 폰 진동에 도달한다. B는 "이번엔 무겁다"는 사용자 판단으로 보류, C는 채널 신뢰 경계 caveat 때문에 학습 보상이 줄어 보류.

**아키텍처 한 바퀴 (M1 = 데몬+PWA 직결, 릴레이는 M2):**
```
[셸 인터셉트(M0에서 메커니즘 확정)] armed + 게이트 명령만 붙잡아 → (Unix socket, 블로킹) → [Host 데몬]
   ※ 평소(비armed)엔 fire-and-forget IPC만 → ≤10ms, 일반 셸 안 멈춤
[Host 데몬 (Rust/tokio)]
  컨텍스트 스냅샷(§31.10) + stale/mismatch 검사 → 위험도 0~100 분해(§31.4, 보수적: alias/eval/미해결변수=상향)
  → 승인 payload(명령+컨텍스트+점수+nonce+approval_id+expires_at+context_hash+device_epoch) 생성
  → best-effort 마스킹(§31.8) → 디바이스 공개키로 암호화 (페어링 시 QR 키 교환, TOFU)
        |
   [M1: Tailscale 직결]  /  [M2: 자체 WebSocket 릴레이 — 채널 토큰 인증, 암호문만 라우팅]
        |
  [PWA (폰)]
  복호화 → 컨텍스트(+stale 경고)+위험도 표시 → WebAuthn(HTTPS origin·등록 필요, 불가 시 PIN) 게이트 → Approve/Reject
  → 서명 토큰(approval_id, nonce) 회신
        |
  [Host 데몬] 서명·nonce·만료·device_epoch·context_hash 재검증 → 위험도 게이트(기본 Medium, High opt-in, Critical 불가)
  → 실행/차단 → 감사 로그(§10.5: 해시된 명령+위험도+디바이스ID+결정+마스킹 컨텍스트, 원문 시크릿 미저장)
  ※ 저하 경로: 폰 timeout/불가 = fail-closed = **차단**(자동 로컬승인 없음, "disarm 후 로컬 실행" 안내)
  ※ 데몬 다운 = hook no-op → 일반 셸 정상 (이건 보안 경계가 아니라 자기-가드레일이므로 허용, Threat Model 참조)
```

## Open Questions

1. **iOS 데모 타깃**: 데모 1차 타깃을 Android/데스크톱 PWA로 못박을지(iOS Web Push는 홈화면 추가 시에만 + 백그라운드 신뢰도 낮음 → iOS 비범위 권장). 또는 Web Push 대신 "PWA 열려있을 때 WebSocket 실시간 + 폴백"으로 단순화.
2. 릴레이 호스팅: 로컬/Tailscale 위 단일 인스턴스 vs 공개 VPS.
3. QR payload 스키마: `relay_url + pairing_code + daemon_pubkey` 정도면 충분한지(폰도 데몬을 인증하는 양방향).
(이전 Open Q4 replay → Premise 6 승격. Open Q5 데몬충돌 → Premise 2 흡수. 프로토콜 primitive는 Next Step 0에서 X25519 + ChaCha20-Poly1305 + Ed25519로 확정 — 질문 아님.)

## Success Criteria

- "한 바퀴"가 진짜로 돈다: preexec hook이 위험 명령을 잡아 데몬에 블로킹 질의 → 폰 PWA가 진동 → 정확한 컨텍스트+위험도 표시 → 승인하면 머신에서 실행, 거부하면 차단.
- **thesis(D) 증명**: payload 생성 시점 컨텍스트가 §31.10 refresh 트리거(cwd≠pwd, branch 변경 등)에 걸리면 stale 경고 표시 또는 승인 보류.
- 릴레이를 장악해도 payload 평문을 못 본다(E2E 검증: 릴레이 로그에 암호문만). 릴레이 접속은 채널 토큰으로 인증된다.
- 페어링/키 교환/AEAD/서명 검증을 직접 구현했다.
- **replay 차단**: 동일 승인 토큰 재전송이 거부된다(nonce 1회용, expires_at 만료).
- **위험도 경계**: 기본값에서 Medium까지만 원격 승인(§30-13), High는 opt-in 시에만, Critical은 어떤 경우에도 폰 승인 불가(거부+로컬 안내).
- **revoke 동작**: 디바이스 revoke 후 그 디바이스의 서명 토큰이 거부된다(lost-device 대응 = revoke + 재페어링).
- 모든 원격 액션이 디바이스 ID와 함께 감사 로그에 남는다.
- PWA 승인 전 WebAuthn(HTTPS origin·등록 필요, 불가 시 PIN) 게이트 통과(§31.3 `require_biometric_for_approval=true` 기본).
- 폰에 표시되는 payload에 **best-effort** Secret/PII 마스킹(§31.8) 적용 — 보장이 아니라 최선 노력(§10.4 "정규식 마스킹은 완전하지 않다"). 인자·env·파일명·branch·heredoc 등에 시크릿이 샐 수 있음을 전제.
- TOCTOU: 승인 후 실행 직전 context_hash 재체크, 드리프트 시 거부+재승인. revoke는 단조 device_epoch로 검증(대기 중 revoke→거부).

## Distribution Plan

- **호스트 데몬**: Rust 단일 바이너리(`cargo build --release`), 학습 단계에선 GitHub Release + 체크섬이면 충분(§14 배포 가이드 흐름 재사용). 자동 업데이트는 범위 밖.
- **PWA**: 정적 호스팅(릴레이와 같은 인스턴스 또는 GitHub Pages). 앱스토어 없음 — URL 열고 "홈 화면에 추가".
- **릴레이**: self-host 단일 인스턴스(Tailscale 위 또는 작은 VPS). 학습용이라 관리형 릴레이는 안 함(§30-12 MVP 제외와 일치).
- CI/CD: 데몬 빌드+테스트는 GitHub Actions 매트릭스(Linux 우선) 재사용. PWA는 정적 배포.

## Threat Model (eng review 추가)

이 빌드는 **"자기 자신용 가드레일"** 이다 — 사용자가 실수로 위험 명령을 실행하는 걸 늦추고 원격에서 한 번 더 확인하게 한다. **악성/탈취된 로컬 유저에 대한 보안 경계가 아니다**: 그런 공격자는 hook 비활성화·다른 셸에서 직접 binary 실행·PATH 변조·데몬 kill로 우회할 수 있다(그래서 데몬-다운=일반 셸 동작이 허용된다). 신뢰 경계는 "정직한 사용자 ↔ 네트워크/릴레이"이며, E2E는 payload 평문을 보호하지만 메타데이터(타이밍·디바이스ID·온라인 상태·IP)는 보호하지 않는다.

## Milestones (eng review 재배열)

- **M0 — 인터셉트 제어점 증명 (착수 전 spike).** zsh ZLE accept-line 위젯 / bash DEBUG·command-not-found / 래퍼 함수 중 무엇으로 "명령을 실행 전 붙잡아 데몬에 묻고 결과로 실행/차단"이 셸별로 실제 되는지 크립토 없이 로컬로 증명. **이게 가장 큰 미지수이므로 최우선.** (codex: preexec 반환값 차단은 비이식적)
- **M0.5 — 와이어 프로토콜 스펙.** 1페이지가 아니라 제대로: **핸드셰이크는 Noise(예: Noise_XX) 검증 패턴 사용**(AKE 직접 굴리지 않음), hand-roll은 앱 레이어(payload·nonce·approval_id·expires_at·device_epoch·게이트)에 한정. identity/transcript 바인딩·key confirmation·replay window·메시지 순서·디바이스 교체 명시. primitive: X25519+ChaCha20-Poly1305+Ed25519(검증 라이브러리).
- **M1 — 데몬+PWA 직결 한 바퀴 (릴레이 없음, Tailscale).**
  - 컨텍스트 스냅샷 모듈(§31.10) + stale/mismatch 검사 + **context_hash(env allowlist 해시·realpath 타깃 포함, "전부는 못 묶음" 명시)**. 모바일 없이 단독 검증.
  - 위험도 0~100 엔진(§31.4) — **shared-core 크레이트**로 hook-side와 데몬 공유. 보수적 분류(alias/eval/명령치환/미해결변수=상향, 모호→높게, critical 패턴 하드블록).
  - armed 상태머신(arm/disarm) + fire-and-forget hook(평소 ≤10ms) + 게이트 시에만 중작업.
  - 페어링 TOFU(QR=daemon_pubkey 신뢰앵커, pairing_code로 폰 인증, 동시 페어링 거부) + best-effort 마스킹.
  - 승인 왕복: 빌드→암호화→서명·nonce·만료·**device_epoch**·**context_hash 실행 직전 재체크**→실행/차단.
  - 위험도 게이트(Medium 기본, High opt-in, Critical 불가) + revoke(`ai remote revoke`, 단조 epoch, 대기 중 revoke→거부) + 감사 로그.
  - 저하: 폰 timeout/불가=차단(자동 로컬승인 없음), 데몬 다운=hook no-op.
  - PWA: 복호화→컨텍스트(+stale)+위험도→WebAuthn(불가 시 PIN)→서명 회신. WebCrypto non-extractable.
  - **보안-핵심 테스트 게이트**: 크립토·replay·TOCTOU·revoke·Critical차단 = proptest + known-answer + 음성 케이스 통과해야 데모.
- **M2 — 릴레이 삽입.** "암호문만 중계" untrusted-intermediary E2E 증명. 채널 토큰 접속 인증. 릴레이 로그에 암호문만임을 테스트로 증명. 메타데이터 누출은 수용(Threat Model).
- **데모.** "arm → 외출 → 진동 → (stale면 경고) → WebAuthn → 승인 → context_hash OK → 실행" 한 바퀴 녹화.

## What I noticed about how you think

- D2에서 제가 추천한 단일 선택(A)을 받지 않고 **C+D를 직접 결합**했다. 그건 단순 변덕이 아니라 "원격 승인의 가치는 컨텍스트 정확도에서 나온다"는 연결을 본 거다 — 두 기능의 의존 관계를 스스로 묶었다.

- 17개 설계 문서 전체가 "안전·일관성"에 무게를 싣고 있다 — 당신은 "멋"보다 "틀리지 않음"을 먼저 보는 사람이다. 그래서 D5에서 결국 정통성(E2E)과 도달 속도(PWA)를 절충하는 가운데 경로 A로 수렴한 게 일관된다.
