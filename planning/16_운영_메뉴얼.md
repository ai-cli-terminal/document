## 1. 모니터링

### 1.1 로컬 진단 (`ai doctor`)

각 단말에서 상태를 점검하는 1차 도구입니다. 외부로 아무것도 전송하지 않습니다.

```bash
ai doctor [--ai] [--sandbox] [--policy] [--context] [--guardrails] [--metrics]
```

```text
AI Terminal Doctor
Shell:   bash 5.2 detected
PTY:     OK
AI provider: remote provider reachable
Sandbox: container backend available / user namespace enabled / no-new-privileges enabled
Policy:  active profile: balanced
Masking: secret rules 18 loaded / pii rules 7 loaded
Context: cwd synchronized / git branch synchronized
```

### 1.2 로컬 메트릭 (`ai doctor --metrics`)

외부 전송 없이 단말 자체의 운영 메트릭을 관찰합니다. opt-in OpenTelemetry를 켜기 전, 또는 켜지 않고도 자체 점검할 때 사용합니다.

### 1.3 OpenTelemetry 메트릭 (opt-in, §29.5)

조직 차원의 SLO 관리가 필요하면 OTLP 익스포트를 활성화합니다. **기본은 비활성이며, 활성화해도 명령 내용은 절대 전송하지 않고 메트릭/트레이스만 보냅니다.**

```toml
[telemetry]
enabled = false               # 기본 비활성, 조직 정책으로 강제 가능
otlp_endpoint = ""
metrics_only = true
no_command_content = true     # 활성화해도 항상 강제
```

수집 메트릭 목록(예):

| 메트릭 | 의미 |
|---|---|
| `ai.request.latency` | AI 요청 지연 (connect / first_token / total) |
| `ai.request.timeout.count` | AI 요청 타임아웃 횟수 |
| `ai.request.cancelled.count` | 사용자 취소(Ctrl+C) 횟수 |
| `cmd.exec.latency` / `cmd.exec.failure.rate` | 명령 실행 지연 / 실패율 |
| `ai.command.accepted.rate` / `ai.command.edited.rate` | 제안 수락률 / 수정률(품질 프록시) |
| `guardrails.triggered.count` | Guardrail 발동 횟수 |
| `masking.applied.count` / `masking.failure.count` | 마스킹 적용 / 실패 횟수 |
| `tokens.input` / `tokens.output` / `cost.estimated` | 토큰 사용량 / 예상 비용 |

> 명령 내용·Secret·PII는 메트릭 경로로도 전송되지 않습니다.

## 2. 알림 · 임계치

조직 운영에서 주시해야 할 핵심 신호와 임계치입니다. 채널(Slack/Webhook/이메일 등)은 조직 표준에 맞춰 연동합니다. <!-- TODO: 알림 채널 구체 연동 방식은 원본 미정의 -->

| 신호 | 임계치 / 기준 | 대응 |
|---|---|---|
| 예산 초과 | 세션 $2 / 월 $30의 **80% 경고**, **100% 원격 AI 차단** | 예산 상향 또는 로컬 AI 전환 안내 |
| Guardrail 발동 | `guardrails.triggered.count` 증가 | 차단된 명령·incident 요약 확인 |
| 마스킹 실패 | `masking.failure.count` > 0 | 원격 전송 차단(fail-closed) 확인, 규칙 점검 |
| AI 타임아웃 | `ai.request.timeout.count` 급증 | provider 가용성·네트워크 점검 |

예산 임계치 메시지 예시:

```text
Monthly AI budget limit reached. Remote AI requests are blocked.
Switch to local AI mode or increase the configured budget.
```

## 3. 장애 대응 플레이북

핵심 불변식: **AI·샌드박스·마스킹에 문제가 생겨도 일반 셸은 계속 사용 가능**해야 하며, 안전이 확보되지 않으면 **자동 우회 없이 차단**합니다.

### 3.1 증상: AI provider 불가 / 타임아웃

- **동작**: 셸은 정상 유지. AI 요청만 취소되고 프롬프트로 복귀.
- **타임아웃 기준**: 연결 5초 / 첫 토큰 15초 / 전체 60초 / 요약 180초.

```text
AI request timed out. Shell mode remains active.
AI service unavailable. You can continue using the terminal normally.
```

- **대응**: `ai doctor --ai`로 연결 확인. 멀티 프로바이더 폴백이 허용된 경우 secondary provider로 전환되며 사용자에게 어떤 모델이 쓰였는지 표시(§29.10).

### 3.2 증상: 마스킹 실패

- **동작**: 원격 AI 호출을 **차단(fail-closed)**. 절대 마스킹 없이 전송하지 않음.

```text
Sensitive data may be present in the selected context.
Remote AI request blocked by privacy policy.
Use local AI mode or reduce context scope.
```

- **대응**: 컨텍스트 범위 축소 또는 로컬 AI 사용. 마스킹 규칙(§31.8) 점검. 정규식이 못 잡는 고엔트로피 토큰은 휴리스틱으로 추가 탐지하며, 불확실하면 차단합니다.

### 3.3 증상: stale lock (잔여 락)

- **원인**: 세션이 비정상 종료되며 락이 남음.
- **동작**: 다음 실행이 stale 여부를 판정해 복구합니다. 판정 기준: PID 부재 / heartbeat가 TTL 초과 / owner session 종료 / lock 파일 mtime이 TTL 초과.
- **복구 절차**: ① stale 확인 → ② stale metadata를 audit log 기록 → ③ lock 제거 → ④ 재시도.

| Lock | TTL | 용도 |
|---|--:|---|
| db.lock | 10초 | 짧은 DB write |
| usage.lock | 10초 | usage 집계 |
| index.lock | 30분 | 인덱싱 |
| policy.lock | 10초 | 정책 갱신 |
| session.lock | 세션 생존 기간 | 세션 소유권 |

### 3.4 증상: DB busy timeout

- **원인**: 동시 세션이 많아 SQLite 쓰기 경합. WAL 모드 + `busy_timeout = 5000`(5초)로 직렬화.
- **동작**: busy timeout 초과 시 조용히 실패하지 않고 명확한 오류를 표시.
- **대응**: 동시 세션 수 확인. 동시 세션 3+ 예산 공유 등이 상시화되면 Phase 2 데몬 도입을 검토(§30-2).

### 3.5 증상: 샌드박스 실행 실패

- **동작**: 샌드박스 실행이 실패해도 **실제 실행으로 자동 전환하지 않습니다.**

```text
Sandbox execution failed. Reason: container runtime unavailable.
Run directly without sandbox? [y/N]
```

- **대응**: 사용자가 명시적으로 동의할 때만 비샌드박스 실행. 컨테이너 런타임 가용성 점검.

## 4. 보안 운영

### 4.1 정책 프로파일 · 조직 정책 우선순위 (§12.3)

MVP는 로컬 사용자 정책(`balanced`/`paranoid`)을 사용합니다. 조직 정책 중앙 배포는 Phase 3입니다. 우선순위는 다음과 같습니다.

```text
Organization Policy > Team Policy > User Policy > Default Policy
```

```text
~/.config/ai-terminal/policy.d/
  00-org-policy.toml
  10-team-policy.toml
  20-user-policy.toml
```

조직 정책은 `readonly = true`로 적용되어 **사용자가 완화할 수 없습니다.** 예: 원격 AI 차단, 허용 provider를 로컬로 한정, 위험 패턴 차단(`curl * | sh`, `chmod -R 777 /`, `rm -rf /`, `docker run --privileged *`), 감사 강제.

```toml
[organization]
name = "Example Corp"
readonly = true

[ai]
block_remote_ai = true
allowed_providers = ["local"]
mask_pii = true
mask_secrets = true

[audit]
required = true
log_path = "/var/log/ai-terminal/audit.jsonl"
```

> Phase 3 조직 정책은 서명 필수, 공개키는 OS trust store/MDM 배포, version monotonic 증가로 rollback 방지(§30-7).

### 4.2 시크릿 / 키링

- AI provider·MCP 자격 증명 등은 **OS 키링**에 저장하며, 디스크에 평문 저장하지 않습니다(`key_source = "os_keyring"`).
- sudo 비밀번호는 PTY를 통해 사용자가 직접 입력하며, 절대 가로채거나 로그·컨텍스트·AI 요청에 포함하지 않습니다(§29.12).
- `allow_sudo_ai_commands = false`(기본)에서는 AI가 sudo 명령을 제안만 하고 `ai run`으로 직접 실행하지 않습니다.

### 4.3 감사 로그 (§10.5)

중요 명령(파일 삭제/권한 변경, 설정 수정, 패키지 설치/삭제, 서비스 재시작, 배포, DB 마이그레이션, 인프라 변경)은 `audit_events`에 기록됩니다.

```json
{ "timestamp": "2026-06-01T10:30:00+09:00", "user": "local-user", "cwd": "/home/user/app",
  "command": "rm -rf ./build", "source": "ai-generated", "risk_level": "medium",
  "confirmed": true, "exit_code": 0 }
```

- Git 관리 디렉터리에서는 실행 전후 `git diff --stat` / `--name-only`를 자동 기록합니다. **원문 diff 전체는 민감 정보 가능성 때문에 기본 저장하지 않습니다.**
- AI 생성 명령에는 `model_id`·`model_version`·`prompt_template_version`·`temperature`·`context_hash`를 함께 기록해 재현·회귀 추적이 가능합니다(§29.7).

### 4.4 서명 · 업데이트 (§29.11)

- **서명된 바이너리** 배포 + 체크섬 검증. 자동 업데이트는 서명 검증 후에만 적용하고, 다운그레이드 공격 방지를 위해 버전 단조 증가를 강제합니다.
- 플러그인은 manifest에 권한을 선언하고 설치 시 승인받으며, 정책 엔진·Zero-Trust Pipeline을 우회할 수 없습니다.

```toml
[update]
auto_update = "notify"        # off | notify | auto
verify_signature = true
allow_downgrade = false

[plugins.security]
require_manifest = true
require_user_consent = true
sandbox_plugins = true
```

## 5. 데이터 거버넌스 (§29.8)

### 5.1 보존 기간

| 데이터 | 기본 보존 | 비고 |
|---|---|---|
| 세션 로그 | 30일 | 경과분 자동 정리 |
| 명령 히스토리 | 365일 | 장기 보존 |
| undo 백업 | 7일 | TTL 경과 시 정리 |
| 캐시 | 1일 | at-rest 암호화 대상 |

```toml
[security.retention]
session_log_retention_days = 30
history_retention_days = 365
undo_retention_hours = 24
encrypt_at_rest = true
key_source = "os_keyring"
allow_right_to_delete = true
```

> 보존 정책 충돌 시: 사용자 메뉴얼의 undo 7일 안내는 백업 TTL(`backup_ttl_days = 7`, §31.6) 기준입니다. 설정의 `undo_retention_hours`는 보강 설계 예시값이며, **MVP 정본은 §31.6의 7일**입니다.

### 5.2 At-rest 암호화

히스토리 DB·세션 로그·캐시는 OS 키링(또는 사용자 패스프레이즈) 기반 키로 암호화합니다. 키는 디스크에 평문 저장하지 않습니다.

### 5.3 삭제 권리

사용자/조직이 데이터를 영구 삭제할 수 있습니다.

```bash
ai history --purge --before <date>   # 지정일 이전 히스토리 영구 삭제
ai forget <session>                  # 특정 세션 삭제
```

조직 정책으로 강제 보존 또는 강제 삭제를 설정할 수 있습니다.

## 6. 백업 / 복구

핵심 대상은 단일 SQLite DB와 설정 파일입니다.

| 대상 | 경로 | 비고 |
|---|---|---|
| 히스토리 DB | `~/.local/share/ai-terminal/ai-terminal.db` | SQLite WAL. `-wal`·`-shm` 동봉 권장 |
| 세션 로그 | `~/.local/share/ai-terminal/sessions/` | jsonl |
| 설정 | `~/.config/ai-terminal/config.toml` | 정본 설정 |
| 조직/팀/사용자 정책 | `~/.config/ai-terminal/policy.d/` | Phase 3 |

- **백업 절차**: 단말 유휴 시점에 위 경로를 복사. WAL 환경에서는 일관성을 위해 세션 종료 또는 체크포인트 후 백업을 권장합니다. <!-- TODO: 원본에 백업 명령/체크포인트 절차 미정의 -->
- **복구 절차**: DB·설정을 원위치에 복원 후 `ai doctor`로 무결성·정책·마스킹 로딩 상태를 확인합니다. stale lock이 남아도 다음 실행이 복구합니다(§3.3).

### 진단 / 디버그 번들 (§23.2)

장애 분석 자료를 안전하게 추출합니다.

```bash
ai doctor --export-debug-bundle
```

- **포함**: 설정 파일, 정책 프로파일, 최근 에러 로그, AI provider 상태, 샌드박스/PTY 상태, 마스킹된 세션 이벤트 일부.
- **제외**: Secret, PII, 전체 명령 히스토리, 원문 로그, `.env`, private key, credential 파일.

## 7. 플랫폼 Capability 고지

동적 guardrails 기능은 플랫폼마다 차이가 있으므로, **모든 OS에서 동일 기능을 보장하지 않습니다.** 차이를 숨기지 않고 명시 고지합니다.

```bash
ai doctor --guardrails    # 현재 플랫폼에서 가능한 guardrail 표시
```

Baseline(모든 플랫폼 공통): static command analysis, risk scoring, preview/diff, dry-run, timeout, process group termination, confirmation, secret/PII masking, policy enforcement.

| 기능 | Linux | WSL | macOS |
|---|---|---|---|
| Static risk analysis / Preview·diff / Timeout | 지원 | 지원 | 지원 |
| Process group termination | 지원 | 부분 | 지원 |
| File count pre-scan | 지원 | 지원 | 지원 |
| cgroups CPU/mem limit | 지원 | 부분 | 미지원 |
| seccomp / fanotify | 지원 | 제한 | 미지원 |
| inotify | 지원 | 지원 | 미지원 |
| FSEvents | 미지원 | 미지원 | 지원 가능 |
| Docker sandbox | 지원 | Docker Desktop 의존 | Docker Desktop 의존 |
| gVisor / eBPF | Phase 3 | 제한 | 별도 구현 |

> 중요: 미지원 guardrail은 **조용히 실패하지 않고 명시 고지**됩니다. 동적 감시가 제한되는 WSL/macOS에서는 보완책으로 **High 이상 명령의 확인을 강화**합니다(§31.11).
