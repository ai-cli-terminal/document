## 0. 설계 검토 요약 (Review & Analysis)

### 0.1 기존 설계의 강점

세 문서는 누적적으로 다음을 잘 확보했다.

- **"일반 터미널 우선" 원칙이 일관됨.** AI는 보조자이며 최종 실행 권한은 사용자에게 있다는 원칙이 모든 문서를 관통한다. AI 장애가 터미널 장애로 전파되지 않는다는 비기능 요구가 명확하다.
- **보안·프라이버시 설계가 실무 수준.** 위험 명령 분류, Secret/PII 마스킹, Zero-Trust Context Pipeline, 샌드박스 capability drop, 조직 정책 우선순위까지 제품화 직전 수준으로 구체적이다.
- **상태 일관성 문제를 정면으로 다룸.** Context Consistency Manager와 Built-in 명령 동기화는 AI 터미널에서 가장 흔한 실패(잘못된 cwd/컨텍스트 기반 명령 제안)를 구조적으로 방지한다.
- **정적 검사 + 동적 감시 이중화.** Policy Engine(실행 전 정적)과 Execution Guardrails Engine(실행 중 동적)을 분리해 방어 깊이를 확보했다.
- **현실적인 단계 계획.** MVP / MVP+ / Phase 2~4 구분이 명확하고, 무리한 기능(eBPF, Firecracker, multi-agent)을 후순위로 잘 미뤘다.

## 1. 개요

본 시스템은 AI CLI 전용 워크플로와 일반 리눅스 터미널 기능을 하나의 환경에서 제공하는 통합 터미널이다. 개발자·운영자·연구자·파워 유저가 기존 리눅스 명령어를 그대로 사용하면서, AI 기반 명령 생성·설명·디버깅·자동화·로그 분석·코드 보조를 자연스럽게 활용할 수 있도록 설계한다.

핵심 목표는 다음과 같다.

- 일반 리눅스 터미널과 완전 호환되는 명령 실행 환경 제공
- AI CLI 명령을 별도 모드 또는 인라인 방식으로 실행
- 명령어 추천·설명·변환·디버깅을 AI가 보조
- 보안·권한·감사 로그를 고려한 안전한 실행 구조 제공
- 로컬, 원격 서버, 컨테이너, 개발 환경에서 확장 가능하게 설계

설계의 핵심 철학은 **"AI는 명령 실행자가 아니라 의사결정 보조자"** 라는 것이다. 기본 동작은 기존 터미널과 동일하게 유지하고, AI 기능은 명시적 호출·실행 전 설명·위험도 분석·사용자 확인을 중심으로 제공한다.

## 2. 설계 목표

### 2.1 기능 목표

1. 일반 리눅스 명령어 실행
2. AI CLI 전용 명령 실행
3. 자연어 기반 명령어 생성
4. 실행 전 명령어 설명 및 위험도 분석
5. 명령 실행 결과 요약
6. 로그·에러 메시지·스택트레이스 분석
7. 반복 작업 자동화 스크립트 생성
8. 프로젝트 컨텍스트 기반 명령 추천
9. 세션 기록 및 재현 가능한 작업 로그 저장
10. 권한이 필요한 명령어에 대한 확인 절차 제공
11. 파일 변경 명령의 preview / dry-run / diff 제공
12. 실행 전 정적 정책 검사 + 실행 중 동적 가드레일

### 2.2 비기능 목표

- 기존 쉘과의 높은 호환성
- 낮은 입력 지연 시간 (일반 명령 경로와 AI 경로 완전 분리)
- 명령 실행 전후의 안전성 검증
- 민감 정보 보호 (Secret/PII 마스킹 필수)
- 플러그인 기반 확장성
- 로컬 우선 동작 지원
- 오프라인/제한 네트워크 환경에서의 기본 기능 유지
- **AI 기능 실패 시 일반 터미널로 끊김 없이 계속 사용 가능**
- PTY 상태와 AI 컨텍스트의 상시 동기화

## 3. 핵심 설계 원칙

본 문서 전체를 관통하는 20개 원칙이다. 설계·구현·리뷰 시 판단 기준으로 사용한다.

1. 일반 쉘 호환성이 AI 기능보다 우선한다.
2. AI는 명령 실행자가 아니라 보조자이며, 최종 실행 권한은 사용자에게 있다.
3. AI 기능 장애는 터미널 장애로 전파되지 않아야 한다.
4. PTY 상태와 AI 컨텍스트는 항상 동기화되어야 한다.
5. 로컬·SSH·컨테이너·Kubernetes 컨텍스트는 명확히 구분한다.
6. 모든 외부 컨텍스트는 Zero-Trust Pipeline을 통과한다.
7. Secret/PII 마스킹은 원격 AI 호출 전 필수이다.
8. 원격 AI 호출 전 필요한 최소 정보만 전송한다 (Data Minimization).
9. 위험도 판단과 정책 평가는 로컬에서 먼저 수행한다.
10. 파일 변경 명령은 preview·dry-run·diff를 우선 제공한다.
11. AI 생성 명령은 기본적으로 자동 실행하지 않는다.
12. 정책 프로파일로 사용자·조직 보안 수준을 조정한다.
13. 샌드박스는 보조 안전장치이지 완전한 보안 경계가 아니다.
14. 실행 전 정적 정책 검사와 실행 중 동적 가드레일을 함께 사용한다.
15. 로컬 LLM·인덱싱은 터미널 응답성을 저해하지 않아야 한다.
16. 감사 로그는 추적 가능성을 높이되 민감 정보는 저장하지 않는다.
17. MVP에서도 상태 일관성·마스킹·타임아웃·preview·정책 프로파일은 반드시 포함한다.
18. 스킬·MCP 도구·플러그인은 모두 Zero-Trust 데이터 취급과 정책 경계 안에서만 동작한다. 추가 능력이 보안 우회 경로가 되어서는 안 된다.
19. MCP 도구 호출과 스킬 스크립트 실행은 일반 쉘 명령과 동일한 위험도 분류·실행 전 확인·감사를 받는다. 부작용을 일으키는 도구는 자동 실행하지 않는다.
20. 원격 제어는 명시적 디바이스 페어링·최소 권한·E2E 암호화를 기본으로 하며, High·Critical 명령의 원격 승인은 기본 차단한다(Low/Medium만 opt-in, §30-13).

## 4. 사용자 시나리오

### 4.1 일반 터미널 사용

```bash
ls -al
cd ~/projects/app
git status
docker ps
```

AI는 사용자가 호출하지 않는 한 개입하지 않는다.

### 4.2 AI에게 명령어 생성 요청

```bash
ai "현재 디렉터리에서 100MB 이상 파일 찾아줘"
```

AI 제안:

```bash
find . -type f -size +100M -exec ls -lh {} \;
```

기본 정책에서는 즉시 실행하지 않고 설명과 함께 사용자 확인을 요청한다.

### 4.3 에러 분석

```bash
ai explain last-error
```

직전 명령어·종료 코드·stderr·현재 디렉터리·관련 파일 정보를 바탕으로 원인과 해결 방법을 제안한다.

### 4.4 위험 명령어 보호

```bash
rm -rf /
```

터미널은 실행 전 차단하거나 강력한 확인 절차를 요구하며, AI 모드는 위험성을 설명하고 안전한 대안을 제시한다.

### 4.5 파일 변경 미리보기

```bash
ai preview "모든 .txt 파일의 foo를 bar로 바꿔줘"
```

임시 복사본에서 실행해 diff를 보여준 뒤 적용 여부를 확인한다.

### 4.6 원격/컨테이너 컨텍스트 작업

```bash
[ssh:prod-web-01:/var/www/app]$ ai "nginx 500 에러 원인 분석해줘"
```

프롬프트가 실행 컨텍스트를 명시하고, AI 컨텍스트도 원격 환경 기준으로 동기화된다.

## 5. 전체 아키텍처

기존 5계층 구조에 **Execution Guardrails Engine**과 **Context Consistency Manager**를 추가한다.

```text
+--------------------------------------------------+
|                Terminal UI Layer                 |
|  - Prompt Renderer                               |
|  - Input Handler (Shell / AI Inline / Hybrid)    |
|  - Output Renderer (streaming, backpressure)     |
|  - AI Suggestion Panel / Visual Preview Pane     |
|  - AI Request Interrupt Handler (Ctrl+C)         |
+-------------------------+------------------------+
```

리모트 접근은 위 5계층을 **가로지르는 별도 게이트웨이**로 둔다(§28). 호스트의 Remote Access Gateway(데몬)가 세션 출력 미러링·확인 승인·알림을 클라이언트(앱/웹/모바일)와 E2E 암호화 채널로 중계한다.

핵심 데이터 흐름 두 가지:

- **일반 쉘 경로**: `Input → Classifier → Policy Engine → PTY 실행 → (Guardrails 동적 감시) → Output` — AI 계층을 거치지 않아 지연이 최소.
- **AI 경로**: `Input → Classifier → Context Consistency 동기화 → Zero-Trust Pipeline → (Skill 매칭/주입) → Token Window Manager → Model Gateway → Agent Pipeline → (MCP 도구 호출 게이트) → 제안/설명/preview → 사용자 확인 → (선택 시) PTY 실행`.

## 32. 결론

본 터미널은 일반 리눅스 터미널의 안정성과 AI CLI의 생산성을 결합하는 것을 목표로 한다. 설계의 핵심은 AI가 명령 실행을 대체하는 것이 아니라, 사용자의 의사결정과 작업 이해를 보조하는 것이다. 따라서 기본 동작은 기존 터미널과 동일하게 유지하고, AI 기능은 명시적 호출·실행 전 설명·위험도 분석·사용자 확인을 중심으로 제공한다.

세 차례에 걸친 설계는 (1) 기능 골격, (2) 안정성·프라이버시·preview, (3) 상태 일관성·동적 가드레일·정책 프로파일을 누적했다. 본 통합본은 이를 하나의 일관된 구조로 병합하면서, 실제 제품화에서 드러날 다음 공백을 추가로 메웠다: 셸 통합 메커니즘, 환각 검증, 신뢰도 게이트, undo 계층, 관측성, 캐싱, 모델 버전 관리, 데이터 거버넌스, 동시성, 멀티 프로바이더 라우팅, 공급망 보안, sudo 처리, 비결정성 테스트, 접근성, 위험도 스코어링.

v3.1에서는 세 가지 요구사항을 정식 서브시스템으로 추가했다: **통합 스킬 관리(§26)**, **통합 MCP 관리(§27)**, **앱/웹/모바일 리모트(§28)**. 이 세 기능은 모두 "AI가 가진 능력·취할 수 있는 액션"을 확장하므로, 새로 만든 것이 아니라 기존 위험도 분류·정책 엔진·Zero-Trust 파이프라인·감사 로그 체계를 **재사용**하도록 설계했다. 스킬 콘텐츠와 MCP 도구 결과는 신뢰할 수 없는 데이터로 다루고, 부작용을 일으키는 스킬 스크립트·MCP 도구 호출은 일반 명령과 동일한 확인·감사를 받으며, 원격 제어는 최소 권한·E2E·Critical 원격 승인 금지를 기본으로 한다.

v3.2에서는 구현 착수 전 결정이 필요했던 13개 항목을 모두 확정했다(§30). 핵심 방향은 **"무엇을 MVP에서 하지 않을 것인가"** 이다 — Hook 기반 통합으로 빠르게 시작하되 데몬·MCP·원격 릴레이·조직 정책 배포는 제외하고, 보안의 핵심인 마스킹·preview·정책 프로파일·context sync·timeout은 포함하며, 로컬 LLM과 시맨틱 인덱싱은 선택 기능으로 둔다. 모든 provider·skill·tool·remote 기능은 capability와 policy를 통해 점진적으로 확장한다. 이 방식은 구현 난이도를 통제하면서도 장기 아키텍처를 막지 않는다.

v3.3에서는 §30.4의 "반드시 확정" 11개 항목을 구현 가능한 명세로 확정했다(§31): 셸 통합 rc UX, 단일 `ai-terminal.db` SQLite 스키마와 파일 락/stale lock 정책, balanced/paranoid 프로파일의 전체 필드값, 0~100 위험도 스코어링, preview/diff 적용 범위, undo 백업 상한, usage event 스키마, Secret/PII 마스킹 baseline, provider capability map, context sync hook 범위, 플랫폼별 guardrails capability matrix. 각 항목은 MVP 수용 기준을 동반하므로, 설계 문서가 곧 구현·검수 계약으로 기능한다. 이 과정에서 위험도 스케일(소가산 → 0~100)과 저장 DB(history.db → ai-terminal.db)의 잠정안을 정본으로 정렬했다(§0.2).

이를 통해 사용자는 기존 리눅스 워크플로를 유지하면서도 명령어 작성·오류 분석·반복 작업 자동화·로그 해석에서 AI의 도움을 받되, 안전성·프라이버시·재현성·운영성이 제품 수준으로 보장되는 환경을 얻는다.

다음 단계는 §31.12 MVP 진입 승인 체크리스트에 서명하고, §31.13 확정값과 §25.1 MVP+ 범위에 따라 첫 구현(셸 Hook + SQLite 스토리지 + 위험도 엔진 + 마스킹 파이프라인)을 시작하는 것이다.
